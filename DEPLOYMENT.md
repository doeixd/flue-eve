## Admission

The compat-server uses `resolveAdmission()` or `resolveAdmissionFromRuntime()` to create the adapter that submits user turns to Flue. The adapter is selected by the following preference order:

| Priority | Adapter | Condition |
|----------|---------|-----------|
| 1 | **In-process** (`createInProcessAdmission`) | Flue agent co-located in the same Node process; requires `@flue/runtime` installed |
| 2 | **Loopback** (`createLoopbackAdmission`) | `FLUE_BASE_URL` environment variable set to a running Flue HTTP server |
| 3 | **Mock** (`createMockAdmission`) | No other adapter available; deterministic replies, no LLM |

The sidecar generated by `flue-eve init` uses `resolveAdmissionFromRuntime()` which implements this preference order automatically:

```ts
// src/flue-eve-shim.ts (generated)
import { resolveAdmissionFromRuntime } from "flue-eve/server";

function resolveEveAdmission() {
  return resolveAdmissionFromRuntime("assistant", {
    flueBaseUrl: process.env.FLUE_BASE_URL,
  });
}
```

- **Local dev without Flue:** omit `FLUE_BASE_URL` â€” mock admission returns deterministic responses for UI testing.
- **Local dev with Flue:** start `flue dev` (or `pnpm flue:dev`); set `FLUE_BASE_URL=http://127.0.0.1:3583` for loopback admission.
- **Production (co-located):** In-process admission is used automatically when the Flue agent module is loaded in the same process. No HTTP loopback needed, no `FLUE_BASE_URL` required.
- **Split deploy:** Set `FLUE_BASE_URL` or a custom `FLUE_AGENT_URL` to point to the remote agent server.

## Mode A â€” same-origin reverse proxy (recommended)

Browser and Eve API share one origin. No CORS configuration required.

```text
Browser â†’ https://app.example.com
           /           â†’ static UI (Vite build)
           /eve/v1/*   â†’ Flue Node server (compat mount)
```

### Build and run (Node)

```bash
vp install
vp exec -- pnpm -r run build
vp exec -- flue build --target node   # in your Flue project root
NODE_ENV=production \
EVE_AUTH_BEARER="$(openssl rand -hex 32)" \
EVE_JOURNAL_PERSISTENCE=sqlite:./data/eve-sessions.db \
FLUE_BASE_URL=http://127.0.0.1:3583 \
node dist/server.mjs
```

Mount the sidecar in `src/app.ts` (generated once by the Vite plugin):

```ts
import { mountEveCompat } from "./flue-eve-shim.js";

mountEveCompat(app);
```

### Reverse proxy (Caddy example)

```caddy
app.example.com {
  handle /eve/v1/* {
    reverse_proxy 127.0.0.1:3583
  }
  handle {
    root * /var/www/app
    try_files {path} /index.html
    file_server
  }
}
```

### Client configuration

`useEveAgent()` with **no `host`** â€” requests go to `/eve/v1` on the same origin.

Pass the bearer token from your app shell (never commit it):

```ts
useEveAgent({
  auth: { bearer: import.meta.env.VITE_EVE_BEARER },
});
```

## Mode B â€” split origins (CORS)

UI and API on different hosts. Requires explicit CORS on the Flue `app.ts` and `host` on the client.

```ts
// Client
useEveAgent({ host: "https://api.example.com" });

// Flue app.ts â€” use the compat-server CORS helper
import { createEveCorsMiddleware } from "@flue-eve/compat-server";

app.use("*", createEveCorsMiddleware({ origin: "https://www.example.com" }));
```

Cookie-based auth is unlikely to work across origins; use bearer tokens.

## Environment variables

| Variable | Where | Purpose |
|----------|-------|---------|
| `NODE_ENV=production` | Node | Enables fail-closed auth when `EVE_AUTH_BEARER` is unset |
| `EVE_AUTH_BEARER` | Node / Worker | Shared secret for `Authorization: Bearer â€¦` on `/eve/v1/*` |
| `EVE_JOURNAL_PERSISTENCE` | Node | `memory`, `sqlite:path`, `redis://â€¦`, or a directory for file-backed journals |
| `FLUE_BASE_URL` | Node | Loopback admission to a running `flue dev` / agent server (e.g. `http://127.0.0.1:3583`) |
| `EVE_AGENT_NAME` | Worker | Agent name exposed in `/eve/v1/health` (default `assistant`) |
| `OTEL_SDK_DISABLED` | Node / Worker | Set to `true` to disable OpenTelemetry auto-instrumentation |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | Node / Worker | OTLP exporter endpoint (e.g. `http://localhost:4318`) |
| `OTEL_SERVICE_NAME` | Node / Worker | Service name for traces (default `flue-eve`) |

**Admission resolution** (`resolveAdmissionFromRuntime`): in-process (preferred when co-located) â†’ `FLUE_BASE_URL` â†’ mock.

### Journal persistence (`EVE_JOURNAL_PERSISTENCE`)

| Value | Backend | Notes |
|-------|---------|-------|
| unset / `none` | In-memory | Sessions lost on restart |
| `memory` | Process `Map` | Dev / single-process with reload |
| `sqlite:./path.db` | Node `node:sqlite` | Requires Node 22+ |
| `redis://host:6379` | Redis via optional `ioredis` peer | Horizontal scale |
| `/path/to/dir` | JSON files per session | Simple single-node deploy |

Wire in the sidecar:

```ts
import { resolveJournalPersistence, resolveEveProductionOptions } from "@flue-eve/compat-server";

eveCompat({
  auth: resolveEveProductionOptions().auth,
  persistence: resolveJournalPersistence(),
  // ...
});
```

### Auth policy

| Context | Default |
|---------|---------|
| `NODE_ENV=production`, no bearer | **Fail closed** â€” all `/eve/v1/*` except health return 401 |
| Local dev | `local-dev` â€” open routes |
| `auth: "none"` | Explicit public demo (Workers default when no bearer) |
| `EVE_AUTH_BEARER` set | Bearer required on session routes |

`GET /eve/v1/health` remains public.

### OpenTelemetry correlation (optional)

The compat-server emits OpenTelemetry spans when `@opentelemetry/api` is installed (optional peer). No mandatory instrumentation dependency â€” spans are no-ops when the peer is absent.

**Span attributes:**

| Span name | Attributes |
|-----------|------------|
| `flue-eve.admission` | `eve.session_id`, `eve.agent_name`, `eve.is_first_turn`, `flue.submission_id` |
| `flue-eve.stream_mapping` | `eve.session_id`, `eve.agent_name`, `eve.stream_event_count` |

**Span lifecycle:**
- `flue-eve.admission`: spans the Flue agent admission call (submission â†’ first terminal event). Records `submission_id` when present.
- `flue-eve.stream_mapping`: spans the Flue event â†’ Eve event mapping within `runTurn`. Fires at turn end regardless of outcome.

Configure via standard `OTEL_*` environment variables (see [OpenTelemetry docs](https://opentelemetry.io/docs/concepts/sdk-configuration/)).

## Cloudflare Workers (M7)

Workers cannot HTTP-loopback to themselves. `createEveWorkerApp` defaults to **mock admission** (deterministic replies, no LLM). Choose an admission strategy based on your deployment goal:

| Option | When to use | Maturity |
|--------|-------------|----------|
| **A. Mock admission** (default) | Contract demos, UI testing, CI smoke | **Shipped** â€” `createMockAdmission()` |
| **B. In-process Flue admission** | Co-located Flue agent on Worker | **M8a** â€” Node only; Worker path pending stable Flue Worker API |
| **C. Service Binding** | Eve compat Worker + separate agent Worker | **M8a spike** (not yet production-ready) |
| **D. External agent URL** | Split deploy: compat Worker + remote Flue HTTP | Document only â€” use `FLUE_BASE_URL` with a public endpoint |

For production chat with a real model on Workers, deploy the Flue agent on Node (Mode A/B) and point the Worker at it via `FLUE_BASE_URL`. In-process Worker admission ([M8a](PLAN.md#m8a--in-process-admission)) is the long-term target.

## Railway / Fly quick checklist

1. Build Flue for Node (`flue build --target node`).
2. Set `EVE_AUTH_BEARER`, `EVE_JOURNAL_PERSISTENCE`, and LLM API keys.
3. No `FLUE_BASE_URL` needed when the Flue agent is co-located (in-process admission is automatic).
4. Put a reverse proxy or platform ingress in front â€” route `/eve/v1` to the Flue server.
5. Deploy static UI to the same hostname (Mode A).
6. Run `pnpm smoke` against the public URL before go-live.

## Known limitations (v1)

- **In-process Flue admission** on Node is stable via `resolveAdmissionFromRuntime()`; falls back to loopback (`FLUE_BASE_URL`) or mock automatically when the Flue agent module is unavailable locally.
- **Cloudflare Workers** use mock admission by default; in-process Worker admission and Service Binding are M8a targets (see admission matrix above).
- **Live MCP OAuth** against a real MCP server requires Flue stream events + optional `@vercel/connect` â€” mock/e2e paths are tested; full live MCP 401 is opt-in.
- **Multi-agent routing**, Eve platform channels, and workflow-run shims are out of scope (see `PLAN.md` Â§20).
- **Per-turn `continuationToken` rotation** is not implemented (stable token per session in v1).
